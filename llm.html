LLM, from First Principles

You paste a half-formed question into a chat box and, within a second, receive fluent paragraphs that sound like a careful colleague. Ask for a citation and it may invent a plausible-looking paper that never existed. Ask it to count the letters in a word and it can stumble while still sounding sure. The same system drafts legal-sounding prose, explains quantum mechanics in metaphor, and then quietly fabricates a meeting that never happened. The friction is not that the machine is “stupid”; it is that fluency and truth-tracking have come apart in a way everyday conversation never prepared you for.

Without a first-principles map, every impressive answer becomes either magic or scam—and you lose the ability to decide where to trust, what to verify, and which layer failed when something goes wrong. Literacy here is agency: you stop negotiating with a ghost and start operating a constrained prediction machine—choosing prompts, checking claims, and knowing when the bottleneck is tokenization, context, sampling, or the training objective itself.

My aim with this article is to rebuild the mechanisms from constraints, introducing jargon only when needed, following the story of problems that forced each next layer.

What this tour will and will not do

I will rebuild the constraint chain from discrete text to preference-shaped chat—in invention order—keep each major step to a few new parts, use interactive toys only where a distribution must be felt, and end with judgment rules that locate failures by layer. I will not ship a prompt cookbook, prophesy timelines, or pretend preference tuning equals truthfulness.

The path of problems

  1. How does typed language become something a machine can compute at all?
  2. What training pressure turns that representation into useful predictions?
  3. How can a late word feel the pull of an early instruction inside one finite window?
  4. How does a cloud of scores become one spoken reply—and why regenerating changes it?
  5. Why does a completion engine start acting like a general task interface—and then like a chat assistant?
  6. When fluency lies, which layer do you interrogate first?

Three everyday mysteries stay in the pocket: invented citations that sound perfect; regenerating that changes wording but keeps the gist; early instructions ignored in a huge paste. Each pays off when its layer arrives.

Language has to become numbers first

You type a sentence into a chat box and the reply looks like language—but underneath, the machine is not holding letters or meanings in the way a person does. A neural network is a stack of matrix multiplications: every input must be a fixed-size vector of real numbers, and every operation expects tensors with predictable shapes. Raw chat text is discrete symbols—Unicode characters, whitespace, punctuation—none of which can be fed directly into a weight matrix. The first architectural failure is therefore blunt: machines cannot multiply "words." Whatever comes next in the pipeline must turn strings into numbers indexed by a finite list.

Token vocabulary

The first move is to stop treating the keyboard stream as the unit of computation. Instead, the model defines a token vocabulary: a closed list of string pieces the tokenizer can emit. A token might be a whole common word (" the"), a fragment ("ing"), or a single character for rare spellings. The vocabulary size is fixed at training time—often tens of thousands of entries—so the model never sees an out-of-vocabulary symbol at runtime; unknown text is simply cut into known pieces. This is easy to misread: tokens are reusable string pieces, not ideas. "bank" as one token does not carry river-vs-finance sense; it is just an indexable substring that often appears in training text.

Integer ids

Each vocabulary entry is assigned an integer id—a plain counting number from 0 to V−1. The string "Hello" might become [15496] if the whole word is one token, or [9906, 995] if split differently; the model never sees the letters, only the id sequence. A chat turn is therefore stored as token ids: a one-dimensional array of integers, one per token, in order. Padding and batching wrap these arrays into matrices, but the semantic commitment is already made—language has been discretized into a finite alphabet of ids before any "understanding" could even be hypothesized.

Embedding lookup

Matrix machines still need vectors, not bare integers. An embedding lookup table is a matrix E of shape [V, d]: row i is the d-dimensional vector assigned to id i. Forward pass for a single id is indexing: vector = E[id]. For a sequence of ids, the lookup returns a sequence of vectors—one row per token—that downstream layers can mix. Crucially, this is not retrieval from a curated encyclopedia: embedding lookup is a lookup table, not a library of truths. Rows start random and are shaped entirely by whatever training pressure comes later; at this layer there is no guarantee that similar ids sit near each other in vector space.

Reconstruction: from string to tensor

Take the prompt fragment "The cat sat". Suppose the tokenizer maps it to three ids:

| Token piece | Integer id |

|-------------|------------|

| "The" | 464 |

| " cat" | 3797 |

| " sat" | 3331 |

The embedding table (say d = 4 for illustration) might return:

E[464]  → [ 0.12, -0.05,  0.33,  0.08]
E[3797] → [-0.21,  0.44,  0.02, -0.19]
E[3331] → [ 0.07,  0.11, -0.28,  0.41]

Stacked, the model receives a 3 × 4 matrix—three vectors in sequence order. No Unicode, no syntax tree, no propositional content—just indexed rows ready for arithmetic. Neural machine translation systems established this id-then-embed pipeline for sequence models (Bahdanau et al., 2014); large-scale language models such as GPT-2 use the same interface at the front door, only with a wider vocabulary and longer contexts (Radford et al., 2019).

Exit proposition

Language enters the model as a sequence of token ids mapped to vectors—not as thoughts.

Those vectors now exist, but nothing in this step yet says which id should follow " sat". The rows of E could drift arbitrarily unless a training pressure ties them to the statistical structure of real text. The next failure is immediate: those vectors still need a pressure that says which id should come next.

The pressure is "what token comes next?"

Chapter 1 ended with floating-point rows indexed by token ids—differentiable, batchable, but semantically unconstrained. You could permute the embedding table at random and, without a training objective, nothing in the architecture would object. The failure is not "the model lacks world knowledge"; it is narrower: without a local objective, embeddings have no reason to arrange themselves for open-ended language use. The entire modern language-model stack is built around one answer to that gap—not a suite of task heads, not a truth criterion, but a single conditional guessing game repeated at every position.

Context as conditioner

Fix attention on one position in a token-id sequence. Everything to its left (in the usual left-to-right setup) is context: the prefix the model may condition on when scoring what comes next. "The cat sat on the" is context; the model's job at the following slot is to allocate probability mass over the entire vocabulary for the next piece. Nothing in the loss asks whether the prefix is a question, a command, or a citation; the only commitment is that earlier ids influence the score vector at the current step. This context as conditioner framing is what lets one stack serve translation, summarization, and chat-like continuation without swapping modules—the format of the prefix changes which ids are likely, but the mechanism is always "given what came before, score the next id."

Next-token distribution

The output of each forward pass at position t is a next-token distribution: a vector of length V whose entries are non-negative and sum to one. Index j is read as the model's assigned probability for vocabulary id j immediately after the prefix ending at t. In practice the network emits raw logits (unnormalized scores); a softmax turns them into probabilities. At "The cat sat on the", mass might concentrate on ids for " mat", " floor", " couch"—not because the model "knows furniture," but because those continuations appeared often under similar prefixes in training text. Local likelihood is not a global truth score. High probability means "this id often followed similar contexts in the corpus," not "this statement is correct in the world."

Cross-entropy on raw text

Training supplies the missing pressure by showing billions of real continuations. For each position, the correct next id from the document is treated as the target class; cross-entropy on raw text penalizes probability assigned to every wrong id and rewards probability on the actual next token. Averaged over positions and documents, gradient descent nudges embeddings and all upstream weights so that likely training continuations receive likelihood. No human labels mark "helpful" or "factual" at this stage—the supervision signal is literally the next character or token cut from the same web page, book, or forum thread. Bengio and colleagues formalized learning a conditional distribution over words from context (Bengio et al., 2003); GPT-2 scales the same recipe: one transformer, one softmax head, one objective over crawled text (Radford et al., 2019). Broad competence can emerge from one objective without a per-task module—not because the loss encodes every skill explicitly, but because many skills are compressible into "predict the continuation."

Reconstruction: one step of the loss

Consider a toy vocabulary V = 5 with ids {the=0, cat=1, sat=2, on=3, mat=4}. After processing prefix ids [0, 1, 2, 3] ("the cat sat on"), suppose logits at the next slot are:

| id | token | logit | softmax p |

|----|-------|-------|-----------|

| 0 | the | −1.0 | 0.02 |

| 1 | cat | −0.5 | 0.03 |

| 2 | sat | −2.0 | 0.01 |

| 3 | on | −1.5 | 0.02 |

| 4 | mat | +3.0 | 0.92 |

The training document continues with id 4 (mat). Cross-entropy loss at this position is -log(0.92) ≈ 0.08—small, because the model already agrees with the corpus. If the true next id were 1 (cat) instead, the same logits would incur -log(0.03) ≈ 3.5, a large penalty driving weights to spread mass differently. Repeat this for every position in every sequence; next-token prediction is the skill being optimized—nothing more exotic at this layer.

Exit proposition

The skill being optimized is next-token prediction from the left context on raw text.

That objective is local: each position scores the immediate next id given everything to its left. Yet a user prompt can place a constraint in the first sentence that must govern a noun twenty tokens later—subject-verb agreement, a defined acronym, an instruction to reply in French. A fixed bag of hand-crafted features around each position cannot reliably route that distant constraint, and compressing the whole prefix into a single recurrent state grows brittle as sequences lengthen. Distant constraints must reach each position inside one finite window—which is exactly the failure the next chapter addresses.

Every position must borrow from the whole window

Next-token prediction tells each position to emit a distribution—but it does not, by itself, say how information from token 3 should influence token 30. Earlier sequence models squeezed the entire prefix through a fixed-size recurrent state updated one step at a time; useful for local syntax, fragile when an instruction or entity name at the start must control a word near the end. Convolutional or fixed-width windows fare worse: anything outside the kernel simply never arrives. The failure is structural: fixed local features and brittle sequential state fail to route early instructions to late words. What is needed is a mechanism that lets every position re-read the whole prefix and decide, with differentiable weights, how much to import from each earlier vector.

Query-key similarity

Split the problem into "what am I looking for?" and "what do each of my neighbors offer?" At position i, the model forms a query vector q_i from the current embedding. Every position j in the context (including i itself) exposes a key vector k_j. A scalar score is the dot product q_i · k_j—large when the two vectors align in direction, small when they do not. Intuitively, the query asks a soft question ("who matches this syntactic role?" "where was the subject introduced?") and each key advertises whether its token is relevant. These scores are not discrete lookups: attention is a soft mixture, not a hard fact pointer. No single key "wins" by fiat; all positions remain candidates with different weights.

Softmax weights

Raw dot products can span a wide range, so the model applies softmax weights across positions for each query: exponentiate scores, divide by their sum, obtain a vector that is non-negative and sums to one. Position j's weight α_{ij} answers "what fraction of my update should come from token j?" If the query at a verb aligns with the key at the sentence subject, that subject's index gets mass; if an early instruction token shares a pattern with a later slot, mass can flow backward across the window. Temperature of the softmax is learned indirectly through key/query scaling; sharp peaks mean "listen mostly here," flat peaks mean "blend many sources." Crucially, the weights are recomputed at every layer and every head—there is no permanent edge labeled "refers-to."

Value mixture

Keys decide how much to import; value vectors v_j decide what gets imported. Each position carries a value vector parallel to its key. The updated representation at i is the value mixture: Σ_j α_{ij} v_j, a weighted average of value vectors from the context window. After this self-attention mix, position i's vector is rebuilt from the whole prefix, not only from its immediate neighbors or a compressed memory cell. Stacking more layers repeats the recipe: stacking layers stacks mixers; one head is not a knowledge graph. Depth adds successive re-blending passes, not a shelf of stored facts with pointers.

Reconstruction: three positions by hand

Take three tokens—ids for "It", " sat", " quietly"—with simplified 2-D value vectors:

| position | token | value vector v |

|----------|-----------|----------------|

| 0 | It | [1.0, 0.0] |

| 1 | sat | [0.0, 1.0] |

| 2 | quietly | [1.0, 1.0] |

Suppose the query at position 2 (quietly) yields softmax weights [0.1, 0.2, 0.7] over positions 0–2. The new vector at 2 is:

0.1 · [1,0] + 0.2 · [0,1] + 0.7 · [1,1]
= [0.1, 0.0] + [0.0, 0.2] + [0.7, 0.7]
= [0.8, 0.9]

The output is neither any single input row nor a hard copy of position 0—it is a similarity-weighted average determined by query-key alignment. Vaswani et al. introduced this parallel mixing block as the core of the Transformer (Vaswani et al., 2017); earlier attention over encoder states in seq2seq models showed the same soft routing idea in a narrower form (Bahdanau et al., 2014).

Expand simulation — Attention replaces a token's representation with a similarity-weighted average of value vectors from the context.

Exit proposition

Self-attention rebuilds each position as a similarity-weighted average of value vectors from the context.

After mixing, the network still holds a distribution over vocabulary ids at the final position—it has not yet committed to a single spoken continuation. Training optimized likelihood; inference must emit one id at a time, append it, and run again. A scored distribution is not yet a spoken reply—that gap belongs to the sampling loop in the next chapter.

Speech is sampled one token at a time

You hit regenerate and the answer returns with different adjectives, a reordered bullet list, maybe a new opening joke—yet the gist often survives. That everyday mystery is not the model “changing its mind” between beliefs. After self-attention mix, each step still only owns a cloud of scores over the vocabulary. Users need a sequence of readable tokens; the machine must turn that cloud into one irreversible choice, then condition on it. Always taking the single highest score makes text dull and repetitive. Drawing uniformly from the whole vocabulary makes gibberish. The forced middle path is structured randomness over the same conditioned scores.

Autoregressive loop

Generation is a loop, not a one-shot paragraph dump. At step t the model receives the prompt plus every token already emitted, runs the stack of mixers, and produces logits for the next position only. One token is chosen; it is appended; the loop repeats until a stop condition. This is the autoregressive loop described for decoder-style transformers in Attention Is All You Need and used throughout GPT-2: each new id becomes permanent context for everything that follows. There is no separate “planning module” that commits to a finished essay and then types it; local choices accumulate into global shape—including early mistakes that later tokens must paper over.

Softmax sampling

Logits become a next-token distribution via softmax: exponentiate, normalize, obtain probabilities that sum to one. Softmax sampling draws an id from that distribution (or a truncated variant of it). Argmax is the degenerate case that always picks the mode. Nucleus or top-k methods simply zero out the long tail before sampling; they do not add facts. Reconstruction with a toy vocabulary {the, a, cat, sat} and logits [2.0, 0.5, 1.0, -1.0]:

probs ∝ exp(logits)
≈ [0.54, 0.12, 0.20, 0.03]  (after normalize; numbers illustrative)

Draw once: you might get the. Draw again from the same vector: you might get cat. Regeneration is re-rolling this dice at every step—not consulting a second world model.

Temperature sharpness

Products expose a control that reshapes the same logits before softmax. Divide logits by temperature T > 0. When T is small, peaks sharpen and the mode dominates. When T rises, the distribution flattens and lower-probability tokens gain mass. That is temperature sharpness: a reshape of scores, not a knowledge injector. If the model never assigned probability to a correct rare fact, raising T will not invent a reliable citation—it will only make fluent wrongness more varied.

Expand simulation — Raising temperature flattens the next-token distribution, increasing the chance of lower-probability tokens.

Worked contrast on two logits [3.0, 1.0]:

| T | p(token A) | p(token B) |

|---|------------|------------|

| 0.5 | ~0.98 | ~0.02 |

| 1.0 | ~0.88 | ~0.12 |

| 2.0 | ~0.73 | ~0.27 |

Same underlying scores; different sharpness; different sample diversity. Everyday regenerate-with-same-gist sits here: high-probability skeleton tokens recur; decorative low-mass tokens jitter.

Anti-confusion to keep in grain: temperature does not add facts under a “creativity” label. And a sampled token is irreversible context—later steps cannot un-say it without a new generation or an edit outside the loop.

Exit proposition

Replies are sampled left-to-right; temperature reshapes the same scores rather than adding facts.

Document continuation still optimizes for plausible web-like text. Users want something that follows instructions and matches rated helpfulness—that is the next pressure, not another sampling trick.

Prompts become programs; preferences reshape the policy

Autoregressive sampling can continue any prefix—including a half-written blog post, a Python file, or a chat log. That is still document completion. People using chat products want something sharper: show three examples of a format and have the model absorb the pattern; ask for a bulleted plan and get bullets; ask the system to refuse a harmful request and see a refusal style. Those behaviors are not free gifts of softmax. They come from two successive pressures layered on the same engine: making the prompt itself a usable program at scale, then reshaping the policy with human preferences so the default interface feels like an assistant rather than a haunted autocomplete.

In-context pattern completion

When the context window contains a task description and a few input–output exemplars, a sufficiently trained model often continues in the demonstrated format—without a gradient update. Language Models are Few-Shot Learners (GPT-3) names this regime clearly: the prompt supplies the “program,” and in-context pattern completion is still conditioned next-token prediction. Reconstruction (schematic):

Prompt:
  translate English to French
  sea otter => loutre de mer
  cheese => 
Model continues with high mass on: fromage

Nothing in the weight matrices was fine-tuned on that exact triple during the request. The continuation is what the next-token distribution assigns after reading the demonstration tokens. Anti-confusion: this is not “learning without reading the prompt.” The exemplars are the conditioning. Small models often fail the same trick because they never absorbed enough “treat the prompt as a form to complete” statistical patterns during pretraining—not because they lack a separate meta-learning soul.

Instruction fine-tuning

Raw pretraining text is full of documents, not of polite Q&A. Even a strong few-shot model can ignore a bare imperative or meander like a scraped webpage. Instruction fine-tuning adds a supervised stage: show many (instruction, desired response) pairs and continue next-token prediction on the response side. The part inventory stays small—same architecture, same sampling loop—but the distribution shifts toward “when you see an instruction-shaped prefix, emit assistant-shaped continuations.” This is still likelihood training; it does not install a fact checker. It makes the interface of following instructions more probable.

Preference-shaped policy

Humans disagree about which of two valid-looking answers is better: clearer, safer, more concise, less sycophantic. InstructGPT documents the forced next step: collect comparisons, fit a preference signal, and optimize the generative policy against it (after supervised instruction data). Call the result a preference-shaped policy. Product differences in tone, refusal style, and “as an AI…” framing often live here. Anti-confusion locked from the glossary: this is not “aligned to truth.” Preference data rewards what raters prefer under their instructions; fluent falsehoods can still win if they look helpful. Hallucinated citations can survive preference training because local plausibility still scores well unless raters specifically punish it every time.

Everyday stake, kept at the chapter’s open and now closed as mechanism: when one chat product refuses and another completes, or when “Act as…” changes tone, ask whether the preference-shaped policy and instruction data differ—before inventing a moral module inside the tokenizer.

| Layer | What it optimizes | What it does not buy |

|-------|-------------------|----------------------|

| Pretraining LM | next-token prediction on raw text | Instruction manners |

| In-context pattern completion | Continuations consistent with prompt demos | Guaranteed task correctness |

| Instruction fine-tuning | Likely assistant-shaped replies | Truth oracle |

| Preference-shaped policy | Higher rated replies under rater rules | Grounding / world-state checks |

Exit proposition

Scale makes prompt patterns usable; preference training reshapes the interface—neither installs a truth oracle.

Fluent helpful answers can still be false, and the window that holds your instructions remains finite. The next chapter diagnoses those failures by layer instead of by mood.

When fluency lies, name the layer

The entrance mysteries return with names now. A citation that looks perfect but points nowhere is often distributional plausibility succeeding: the model emits token sequences that resemble the bibliography dialect of the web, without an outer check that the paper exists (GPT-3 and InstructGPT both treat untruthful or biased continuations as expected limits of likelihood-plus-preference systems). A huge paste where the first bullet of constraints vanishes is often the finite context window: vanilla self-attention mix is quadratic in length (Attention Is All You Need), products truncate or compress, and early ids simply never enter the mixer. Diagnosing either failure as a single ghostly “AI mood” wastes agency. Name the layer.

Distributional plausibility

Next-token prediction rewards local fit to training-like text. When the true answer is rare, unknown, or never present in the window, the highest-mass continuations can still be grammatical, confident, and wrong. That is hallucination as distributional plausibility—not proof of a deceptive agent with a private world model. Reconstruction: ask for “the 2019 paper by Smith & Lee on underwater basket weaving DOI…” and watch a DOI-shaped string appear. The pieces (year, author pattern, DOI syntax) are high-mass fragments; their conjunction was never verified. Preference-shaped policy can reduce some of these cases when raters punish them, but it does not change the fact that the core loss never required a world-state check.

Finite context window

Every deployed LM attends over a bounded token list. GPT-3’s public configuration used a 2048-token context (Brown et al.); modern products enlarge the budget, yet the budget remains a budget. If your system prompt, retrieved docs, and user paste compete, something drops. Chat UIs that “remember” prior turns are outer systems: summaries, embeddings, databases—not proof that the core mixer holds infinite history. When an early instruction is ignored, inspect truncation and position before accusing disobedience.

Outer verification loop

Truth-tracking is optional infrastructure wrapped around the generative engine: retrieval with citations you open, code execution, unit tests, human review, calculators. Call this the outer verification loop. It is not implied by cross-entropy. The pedagogical move is to stop expecting the LM loss to purchase it. If the task is checkable, check outside the prose. If it is not checkable, treat fluency as a draft, not a verdict.

| Symptom | First layer to interrogate | Weak explanation to reject |

|---------|----------------------------|----------------------------|

| Fake citation / invented quote | distributional plausibility | “It wanted to lie” |

| Early rules ignored in a long paste | finite context window | “It disobeyed” |

| Wording changes on regenerate | autoregressive sampling | “It changed beliefs” |

| Sudden refusal / tone shift by product | preference-shaped policy | “The tokenizer learned morals” |

| Need a correct number or source | outer verification loop | “Higher temperature will find truth” |

Payoff for the foreshadowed mysteries: invented citations → plausibility without checks; regenerating different wording → sampling (prior chapter); ignored early instructions → finite context / mixing budget.

You now stand somewhere on the ladder—token ids, next-token prediction, self-attention mix, autoregressive sampling, in-context pattern completion, preference-shaped policy, or outer verification. The close of the article crystallizes judgment rules you can run without reopening the papers.

Exit proposition

Failures localize to prediction, mixing, sampling, preference, or missing outer checks—not to one “AI mood.”

What remains is to carry a pocket map: when a new everyday mystery arrives, choose the layer first, then decide whether to re-prompt, shorten context, change sampling, switch products, or verify outside the model.

Carry a pocket map, not a vibe

You do not need to recite paper titles to keep agency. You need a map of layers and a habit of asking which layer failed before you negotiate with a ghost.

Judgment rules

  1. Fluent but checkably false → interrogate distributional plausibility first. The loss rewarded locally typical text; it did not require a world-state check. Open the citation, run the code, ask for a source you can click.
  2. Early instructions vanish in a long paste → interrogate the finite context window and what the product truncated or summarized. Re-place the constraints near the end, shorten, or retrieve—do not only raise temperature.
  3. Regenerations diverge in wording but share a skeleton → you are watching autoregressive sampling over the same conditioned distribution. Temperature sharpness changes diversity, not the knowledge inventory.
  4. Tone, refusals, or “as an AI…” framing differ by product → ask about preference-shaped policy and instruction data. That is interface shaping, not proof the tokenizer learned morals.
  5. You need truth, not draft prose → add an outer verification loop. Retrieval, tools, tests, and humans are not optional manners; they are the missing channel the LM loss never bought.

When to descend the ladder

Stay high when the task is brainstorming, rephrasing, or exploring formats—in-context pattern completion and preference-shaped chat are enough. Descend when something breakable is at stake: ids and tokenization when weird splits corrupt code; self-attention mix and window length when constraints drift; sampling when outputs are oddly repetitive or chaotic; preference layer when the product behavior shifted; outer verification whenever a claim must survive contact with the world.

The wrong move is to push harder on the same layer—longer pleading prompts against a truncated window, or “be more creative” against a missing fact.

Life placement

Say aloud which layer you are operating on:

  • putting language into token ids and embedding lookup
  • training/using next-token prediction
  • routing constraints through self-attention mix
  • emitting text via autoregressive sampling
  • steering with in-context pattern completion
  • meeting a preference-shaped policy
  • checking with an outer verification loop

If you cannot name the layer, you are back in vibe territory—and that is exactly when fluent systems reclaim your agency.

The chain in one breath

Discrete text becomes token ids and vectors; next-token prediction supplies the pressure; self-attention mix lets every position borrow from a finite window; sampling turns scores into irreversible speech; scale makes prompts act like programs; preference training reshapes the assistant interface; none of that installs truth—so you verify outside when it matters.

That is the mental model. Use it the next time a citation looks perfect, a regenerate feels like a new opinion, or an early rule quietly disappears from a long paste.
